# Changes Log

Chronological history of changes made to this codebase (post-context.md). Each entry: one concise line — what changed and where.

- 2026-07-18 — Added `context.md` at repo root documenting full architecture, folder structure, backend/frontend patterns, domain model, and known inconsistencies; added this `changes.md` file.
- 2026-07-18 — Fixed `dotnet ef database update` failing with LocalDB startup error: `AppDbContextFactory` (`backend/src/ReplyCart.Infrastructure/Persistence/AppDbContextFactory.cs`) had a hardcoded `(localdb)\mssqllocaldb` connection string used only by EF Core design-time tooling; replaced with the Dockerized SQL Server connection string (matching `appsettings.Development.json`), overridable via `ConnectionStrings__DefaultConnection` env var.
- 2026-07-19 — Fixed `dotnet ef database update` failing with "column/table already exists" errors: `Persistence/Migrations/20260521115712_AddFaviconAndLoaderToStorefrontSettings.cs` had been scaffolded against a stale model snapshot (from the migrations-folder move) and redundantly re-created ~20 columns and 8 tables already added by earlier migrations (`AddAuthProductionFlows`, `AddCouponsReviewsAbandonedCarts`, `AddLandingPageConfig`, `AddPlatformSettings`, `AddAutonomousAi`, `AddCustomerBirthday`, `AddWhatsAppCatalogId`, `AddPaymentGatewayFields`); stripped the duplicate `CreateTable`/`AddColumn`/`CreateIndex` calls (and matching `Down()` entries) from that migration, keeping only its genuinely new work (Cloudflare custom-domain columns on `Tenants`, favicon/loader/GA4-refresh-token columns on `StorefrontSettings`, and legitimate column-type widenings). Final `AppDbContextModelSnapshot.cs` was already correct and untouched — this only fixes the migration path, not the end schema.
- 2026-07-19 — Root-caused a recurrence of the same "column names in each table must be unique" error (`Users.IsTwoFactorEnabled` added twice on `dotnet ef database update`): the previous same-day fix above didn't fully stick — the old `backend/src/ReplyCart.Infrastructure/Migrations/` folder (pre-restructure namespace `ReplyCart.Infrastructure.Migrations`, last touched pre-restructure) was still present alongside `Persistence/Migrations/` and both compiled into the same assembly, so EF Core applied both histories in one chronological sequence: old folder's `20260508120000_AddAuthProductionFlows` added `Users.IsTwoFactorEnabled`/`TotpSecret`, then `Persistence/Migrations/20260521115712_AddFaviconAndLoaderToStorefrontSettings` (a squash migration) tried to add them again. Deleted the entire stale `Infrastructure/Migrations/` folder (18 migration files + its `AppDbContextModelSnapshot.cs`); `Persistence/Migrations/` is now the sole migration history, with `20260521115712_AddFaviconAndLoaderToStorefrontSettings` as its first migration.
- 2026-07-19 — At the user's request, reset EF Core migrations entirely rather than keep patching the old history: deleted every remaining migration file in `backend/src/ReplyCart.Infrastructure/Persistence/Migrations/` (all 21 files, including `AddFaviconAndLoaderToStorefrontSettings`, `AddB2CB2BFeatures`, `AddChatbotTokenUsage`, etc.) — `Persistence/Migrations/` is now empty. **Follow-up required (user to run locally, no `dotnet`/DB access from this environment)**: from `backend/`, run `dotnet ef database drop --project src/ReplyCart.Infrastructure --startup-project src/ReplyCart.Api -f` to drop the local dev DB, then `dotnet ef migrations add InitialCreate --project src/ReplyCart.Infrastructure --startup-project src/ReplyCart.Api` to scaffold one fresh baseline migration from the current model, then `dotnet ef database update --project src/ReplyCart.Infrastructure --startup-project src/ReplyCart.Api` to apply it. Afterwards, regenerate `backend/ReplyCart_Database.sql` (now stale) via `dotnet ef migrations script --project src/ReplyCart.Infrastructure --startup-project src/ReplyCart.Api -o ../ReplyCart_Database.sql -i`. **Production risk**: if this schema was ever deployed to the production Azure SQL instance, its `__EFMigrationsHistory` still has the old migration IDs — do not run `database update` against production with this new squashed history until that table is reconciled by hand (see `context.md` §4.6).
- 2026-07-19 — **Correction**: the squash-to-`InitialCreate` described in the entry above was never committed and the deletion has since been reverted — both `Infrastructure/Migrations/` (18 files) and `Infrastructure/Persistence/Migrations/` (16 files) are back and are the real, current history (34 migrations total). Re-audited all 34 migration `Up()`/`Down()` bodies against each other and against the current `AppDbContext`/entity models (this is the actual fix now in place, done without deleting or squashing anything):
  - Found 17 migrations silently skipped by `dotnet ef database update` because they were hand-written without the `[DbContext]`/`[Migration]` attributes EF requires for discovery (no error — they just never ran). Root-caused: `20260521115712_AddFaviconAndLoaderToStorefrontSettings` (a properly-attributed migration) was scaffolded against a snapshot that predated 9 of those un-attributed migrations, so it redundantly re-`CreateTable`/`AddColumn`'d ~8 tables/25 columns already created by earlier (attributed) migrations — guaranteed "already an object named X" on a fresh DB.
  - Fix: trimmed the duplicate `CreateTable`/`AddColumn`/`CreateIndex` blocks (and matching `Down()` entries) out of `AddFaviconAndLoaderToStorefrontSettings`, keeping only its genuinely new work plus the 3 tables/columns whose original migration stays intentionally unattributed. Added `[DbContext]`/`[Migration]` attributes (minimal `.Designer.cs` stub, matching the existing `AddAuthProductionFlows.Designer.cs` precedent, or inline in the raw-SQL migrations matching the `AddChatbotTokenUsage` precedent) to 11 migrations that turned out to be genuine, non-duplicate prerequisites: `AddCountryLanguage`, `AddBrandColors`, `AutoOutOfStockExistingProducts`, `AddGA4ToStorefrontSettings`, `AddMetaWhatsApp`, `AddMetaTemplateFields`, `AddSeoKeywords`, `AddReturnPolicy`, `AddChatbotClients`, `AddChatbotClientChannels`, `AddStorefrontPages`. Left 6 migrations intentionally un-attributed as fully-superseded duplicates: `AddPlatformSettings`, `AddCustomerBirthday`, `AddWhatsAppCatalogId`, `AddPaymentGatewayFields`, `AddAutonomousAi`, `AddB2CB2BFeatures`. See `context.md` §4.6 for the full breakdown and known follow-ups (lost `PlatformSettings` seed row, stale `AppDbContextModelSnapshot.cs`).
